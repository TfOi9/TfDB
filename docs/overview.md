# TfDB

TfDB 是一款由 TfOi9 自主设计开发的简易数据库系统，基于缓存池和 B+ 树实现，已支持并发和简单的容错。

本项目深受 CMU 15-445 ‘BusTub’ 数据库系统的启发。

## 项目设计概览

本数据库实现了类似于 `std::multimap` 的映射关系维护，但基于文件读写。其中，键和值类型可变，但要求大小固定（是 POD 类型），允许插入重复的键，但是不允许键和值都相同。

数据库中，键和值按照大小关系排序，程序会自动探测键和值有没有重载比较运算符，如有则直接使用，否则将会按照内存哈希值进行比较。考虑到内存哈希有碰撞风险，建议确保键和值有比较运算符。

本数据库系统分为多个层次。最底层是磁盘读写层 `DiskManager` 和日志读写层 `LogManager`，中层是缓存管理层 `BufferManager`，上层为 B+ 树索引层 `BPlusTree`。

本数据库中，所有实际数据均以页（`Page`）为单位写入磁盘，页的大小并不固定，可以自定义修改，以内部存储的键值对数目为单位。

## 项目模块介绍

### `DiskManager` 磁盘读写器

`DiskManager` 是一个顺序、无锁的磁盘读写器，基于 `std::fstream` 实现，支持新分配页、磁盘上页的读写和回收空闲页回收。其中，空间回收机制可选择关闭或打开。如空间回收关闭，回收函数将会是空函数。磁盘分配的空间、读写位置均由 `diskpos_t`（磁盘位置类，是一个 64 位整数）来标识。

`DiskManager` 创建的磁盘文件分为文件头和文件主体：文件头由 `info_len` 个 `FixedInfoType` 构成，第一位是空闲表长度，第二位是 B+ 树根位置，之后的位置均用于空闲表存储。文件主体即为按顺序排列的各个页。

`DiskManager` 在内存中维护了空闲页链表，如有释放页面，则把页位置加入链表。在分配新页时，将会优先使用空闲链表中的页面。

`DiskManager` 默认把空闲页位置存储在文件头的第三个位置及以后，如空闲页数量太多，文件头无法容纳，系统将会创建一个空闲页文件，存储所有空闲页位置。需要注意的是，只有在程序退出时，空闲页才会刷盘。程序启动时，将会把空闲页全部读入内存，在内存中建立空闲页表。

### `BufferManager` 缓存池管理器

`BufferManager` 是一个线程安全的缓存池管理器。所有 B+ 树索引层与磁盘读写层的交互，均需要通过缓存池。缓存池基于 LRU 算法实现，缓存大小可调。

具体地，缓存池在内存中维护了帧（`CacheEntry`），保存近期使用的页，来减少刷盘次数。每个帧保存了页位置、指向内存中页内容的指针、页是否被修改、PIN 计数、LRU 链表迭代器和读写锁。

为保证读写的线程安全，B+ 树与缓存的所有数据交流均被封装为 RAII 的页守卫（`PageGuard`），分为读守卫和写守卫。

以下来具体介绍缓存池的数据流：需要读取页面时，索引层调用接口 `read_page` 接口。缓存池将会在池中搜索是否已经持有该页面，如没有，将会从磁盘中读出该页面并存入缓存。得到页面内容后，将会构造读守卫并返回，同时给该页面的 PIN 计数增加 1。需要写页面时，索引层调用接口 `write_page` 接口。缓存池将会类似地先获取页面，增加 PIN、同时把读写锁升级为独占的写锁，把页面修改字段置为已修改，随后返回写守卫。

上述过程中，如果有任何一步发现缓存池已满，将会立刻从缓存中淘汰一个 PIN 值为 0（即表示此页面不在使用中）的帧，如有修改，将会写回磁盘。目前的缓存置换策略是 LRU。

页守卫析构时，将会自动减少 PIN 值并将读写锁解锁。

### `BPlusTree` B+ 树索引层

`BPlusTree` 是一个线程安全的 B+ 树实现，基于蟹行锁耦合协议。B+ 树所有的读盘、写盘操作均由缓存池管理。

`BPlusTree` 对外接口如下：

- `std::optional<ValueType> find(const KeyType &key);` - 按键查找值；如有多个值，仅返回第一个；
- `void find_all(const KeyType &key, std::vector<ValueType> &vec);` - 按键查找值，返回所有值；
- `void insert(const KeyType &key, const ValueType &val);` - 插入一个键值对，如完全重复，将忽略本次操作；
- `void erase(const KeyType &key, const ValueType &val);` - 删除一个键值对，如不存在，将忽略本次操作。

`BPlusTree` 采用了蟹行锁耦合协议，在 `Context` 字段中持有当前所有的读守卫和写守卫。具体地，各个操作流程如下：

- 查找值时，从根向下依次查找。由于只读，仅需持有当前页的读守卫即可。
- 插入值时，由于很有可能仅需写叶子页，故先按查找值的方法，乐观地查找到叶子。如果可以直接插入，则仅需获取叶子页的写守卫，插入后直接返回。但如果发现需要分裂，则必须重新开始，此时只能悲观地持有路径上所有页的写锁，向下蟹行到叶子，插入并分裂后依次上行，修改父节点，最坏情况下需要创建新根。此外，如果下行时发现某中间节点尚有至少一个空位，则可以直接释放上层所有写锁，因为分裂至多持续到本层就会结束。
- 删除值时，类似地，先乐观读，如失败再进行悲观写，持有所有路径上的写守卫，按 B+ 树再平衡的方法试图从左、右兄弟借孩子，如失败则进行合并，直到回到根。
